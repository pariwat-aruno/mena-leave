backend filter ใบลาให้

  • หัวหน้างาน ใบลา Stage 1 ของ ลูกน้องตัวเอง
  • ฝ่ายบุคคล ใบลา Stage 2 (ทุกใบ) · pending register
  • เจ้าของ ใบลา Stage 3 · Pending_Changes ที่ ADMIN propose

เมื่อกด

  • อนุมัติ → stage นั้น = approved → trigger stage ถัดไป (push Flex หาคนต่อไป)
  • ปฏิเสธ → final_status = rejected → คืนโควตา + แจ้งผู้ลา + ทุกคนที่ผ่านมา

ทางลัดอื่น

  • ในแชท LINE ก็กดอนุมัติได้ผ่าน Flex card ปุ่ม อนุมัติ/ปฏิเสธ (postback)
ใช้ตอน พนักงานทั่วไป เข้าหน้านี้จะไม่เห็นใบลา (filter ว่างเปล่า)
เจ้าของระบบจะตั้งให้คุณเป็น หัวหน้างาน (is_supervisor=TRUE) แล้วผูกลูกน้องให้ก่อน คุณถึงจะเห็นใบลา Stage 1 ที่ต้องอนุมัติ

แท็บทั้งหมด

  • พนักงาน รายชื่อ · approve register · invite + pairing code · ระงับการใช้งาน
  • หัวหน้างาน ผูก subordinate ↔ supervisor (เลือกจาก dropdown)
  • โควตา set quota ต่อพนักงาน (ผ่าน Pending_Changes ถ้า ADMIN — รอ OWNER ยืนยัน)
  • กฎ แก้ LeaveRules (advance notice / max consecutive / doc required)
  • Settings brand · approval levels · default quotas · GPS bypass · etc.
  • Report filter ใบลาตาม date/type/status/user

การ invite พนักงานใหม่

  • กด เชิญพนักงานใหม่ → ระบบสร้าง 6-digit code (TTL 24h)
  • copy ข้อความเชิญ → ส่งให้พนักงานทาง LINE/แชท
  • พนักงาน add เพื่อน OA + เปิด LIFF + ใส่ code 6 หลัก → ผูก line_user_id กับ Users row
ใช้ตอน พนักงานทั่วไป · หัวหน้างาน เข้าหน้านี้จะถูก backend block ขึ้นข้อความ "ไม่มีสิทธิ์"
ฝ่ายบุคคล ใช้บ่อยที่สุด — onboard พนักงานใหม่ · pair หัวหน้างาน · ตั้งโควตา
ADMIN แก้ Settings/Rules/Quota จะถูก lock เป็น Pending_Changes รอ OWNER อนุมัติ (กัน ADMIN แอบแก้ระบบ)
