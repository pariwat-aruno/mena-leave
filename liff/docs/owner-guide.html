สิทธิ์รวมของ OWNER

role = OWNER
  • ทุกอย่างที่พนักงาน + ฝ่ายบุคคล (HR/ADMIN) ทำได้
  • อนุมัติชั้นสุดท้าย (Stage 3) ของใบลาทุกใบ — ใบลาจะ approved ได้ต้องผ่านชั้นนี้
  • อนุมัติ Pending Changes — เมื่อ ADMIN propose แก้ quota/rule/settings/users มันจะถูก queue รอ OWNER ตัดสิน
  • เชิญ OWNER ใหม่ ได้ทันที (I-016) — ADMIN ไม่มีสิทธิ์นี้
  • ตั้งค่า Sheet Settings ทั้งหมด โดยไม่ผ่าน Pending_Changes
  • ลาเอง = auto-approve — ใบลาจะอนุมัติทันที + log ใน Audit_Log

อนุมัติ Pending Changes

admin.html → tab "Pending Changes" (OWNER only)

ทำไมต้องมี Pending_Changes

ป้องกัน ADMIN (HR) แก้ระบบเอง — เช่น เพิ่มโควตาให้ตัวเอง, ปิด GPS เพื่อโกง, เปลี่ยน approval levels, เปลี่ยน supervisor ลูกน้องตัวเอง — การกระทำเหล่านี้จะถูก queue รอ OWNER ยืนยัน

ตัวอย่างที่ trigger Pending_Changes

  • ADMIN set quota รายคน (non-default) → queue
  • ADMIN แก้ LeaveRules → queue
  • ADMIN แก้ Sheet Settings → queue
  • ADMIN เปลี่ยน supervisor pair (Supervisors row) → queue
  • ADMIN approve register ใหม่ + สร้าง default quota → ไม่ queue (ปกติ — ใช้ default จาก Settings)
  • ADMIN ระงับการใช้งานพนักงาน → ไม่ queue (operational action)

เชิญ OWNER ใหม่

admin.html → ปุ่ม "เชิญ OWNER ใหม่"

วิธีเชิญ

  • 1. ใน admin → ปุ่ม เชิญ OWNER ใหม่
  • 2. ระบบสร้าง pairing code 6 หลัก (TTL 24h)
  • 3. copy ข้อความเชิญ → ส่งให้ผู้ที่จะเป็น OWNER ใหม่
  • 4. ผู้นั้นเปิด LINE OA → add friend → เปิด LIFF myid.html
  • 5. ระบบเห็นว่ายังไม่ register → ให้กรอก code 6 หลัก
  • 6. ระบบ link line_user_id กับ Users row (role=OWNER) → status=active

ความสำคัญ

  • OWNER เพิ่ม OWNER ใหม่ได้ทันที ไม่ผ่าน Pending_Changes
  • ADMIN ไม่สามารถเพิ่ม OWNER ได้ — แม้จะ propose ผ่าน Pending_Changes
  • การมี OWNER หลายคน = approve stage 3 ได้เร็วขึ้น (1st OWNER wins)
  • ถ้า OWNER คนเดียวลาออก/ลืม phone → ระบบจะลาทุกใบไม่ผ่าน Stage 3 — ควรมีอย่างน้อย 2 OWNER เสมอ
